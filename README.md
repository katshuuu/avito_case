# Avito Services: Microcategory Splitter

Команда: **2 разработчика (NLP/ML + Backend)**  
Задача: по описанию объявления в вертикали "Ремонт и отделка" определить:
- какие микрокатегории упомянуты (`detectedMcIds`);
- нужно ли делать сплит объявления (`shouldSplit`);
- для каких микрокатегорий создавать дополнительные черновики (`drafts`).

---

## 1. Контекст задачи и требования

Вход: объявление с исходной микрокатегорией (`sourceMcId` / `mcId`) и текстом `description`.  
Нужно находить только **дополнительные** микрокатегории (исходная в расчет не включается).

Ключевые требования:
- **критично**: высокая доля корректно найденных микрокатегорий (precision/recall по `targetDetectedMcIds`);
- **критично**: качественные тексты черновиков (`drafts`);
- допускается **удалённый LLM API** для улучшения формулировок черновиков (без локально развёрнутых моделей);
- корректный и быстрый JSON-формат для интеграции у Авито;
- сравнение категорий проводить по `targetDetectedMcIds`;
- технические поля `caseType` и `split` не использовать;
- в gold-данных соблюдать инвариант:  
  `shouldSplit = true` => `targetDetectedMcIds == targetSplitMcIds`.

**Лимит по времени (инференс без LLM):** полный прогон по `rnc_dataset_markup` на CPU укладывается примерно в **3 секунды** (см. раздел 6). Включение `--llm-drafts` увеличивает время и сетевые зависимости — для продакшена обычно выносят генерацию текста в асинхронную очередь.

---

## 2. Гибридная архитектура: эвристики + опциональный LLM

### Ядро (всегда): словарь + эвристики

Мы используем **детерминированный** словарно-эвристический pipeline для `detectedMcIds` и `shouldSplit`:

- **Контроль и объяснимость**: решение трассируется до фраз словаря и сигналов контекста.
- **Скорость и предсказуемость**: линейный проход по тексту; подходит под жёсткие SLA.
- **Нормализация «битого» ввода**: лёгкая замена похожих латинских букв на кириллицу перед поиском (снижает пропуски на объявлениях с опечатками/латиницей).

### Опционально: удалённый LLM только для текстов черновиков

Файл: `src/project_name/llm_drafts.py`

- Один батч-запрос к Chat Completions API на все черновики объявления.
- Включается флагами окружения `OPENAI_API_KEY` + `ENABLE_LLM_DRAFTS=1` и CLI `--llm-drafts`.
- Классификация и split остаются на эвристиках — **LLM не «ломает» метрики** и не требует GPU.

## 3. Логика решения

### 3.1 Детекция микрокатегорий

Файл: `src/project_name/categorizer.py`, морфология: `src/project_name/morph_detect.py`

1. Нормализуем текст (пробелы, lower case, лёгкая латиница→кириллица).
2. Для каждой микрокатегории ищем вхождения `keyPhrases`:
   - **подстрочно** (как раньше);
   - **по словоформам** — для многословных фраз последовательность нормальных форм слов (`pymorphy2`), для однословных — совпадение леммы токена с леммой ключа.
3. **Фильтр precision:** отбрасываются слишком короткие однословные совпадения (подстрока короче 5 букв, морфология одного слова короче 6 букв), кроме устойчивых коротких терминов (`гкл`, `мдф`, `пвх`, `обои` / `обоев`). Многословные фразы не режутся.
4. Исходная микрокатегория объявления **не попадает** в итоговый `detectedMcIds`.

Зависимость: `pymorphy2` (см. `requirements.txt`). Без неё работает только подстрочный поиск.

### 3.2 Решение о сплите (`shouldSplit`)

Комбинация сигналов:
- **комплексное предложение** («под ключ», цельный текст про «все виды внутренней отделки» и т.п.) → обычно **без split**, если нет контр-сигналов;
- **списки/секции** с дефисами, перечисление услуг блоками → чаще **split**;
- **короткое объявление** со списком услуг через запятую без «под ключ» → чаще **split**;
- **длинное объявление** с явным миксом направлений (сантехника + электрика и т.д.) → **split**;
- **одна** дополнительная микрокатегория: короткое узкое объявление часто **без split**, длинное специализированное — **split** (разделение близко к эталону на кейсах вроде `1002306` vs `1002461`).

### 3.3 Формирование `drafts`

Для каждой split-микрокатегории формируется объект:
- `mcId`
- `mcTitle`
- `text` — шаблон по умолчанию; при `--llm-drafts` и настроенном API — **улучшенный** текст.

Поле `text` может быть пустым — формат остаётся валидным.

## 5. Метрики и правила оценки

Скрипт: `scripts/metrics.py`

Считает:
- `precision_micro`
- `recall_micro`
- `f1_micro`
- `shouldSplit_accuracy`

Особенности:
- сравнение категорий — строго по `targetDetectedMcIds`;
- `caseType` и `split` не используются;
- если `shouldSplit=true` и множества в gold не совпадают, скрипт падает с `ValueError`.

## 6. Результаты на rnc_dataset_markup

Команды:

```bash
python3 scripts/evaluate.py \
  --dict data/examples/microcategories_rnc.json \
  --input rnc_dataset_markup.json \
  --output artifacts/predictions/rnc_predictions.json

python3 scripts/metrics.py \
  --gold rnc_dataset_markup.json \
  --pred artifacts/predictions/rnc_predictions.json
```

Получено (текущая версия эвристик, без LLM в прогоне):
- `precision_micro`: `0.9659`
- `recall_micro`: `0.7046`
- `f1_micro`: `0.8148`
- `shouldSplit_accuracy`: `0.6750`

Скорость:
- полный `evaluate` по всему JSON ~**2.0–2.5 с** на CPU (укладывается в лимит **3 с**);
- `metrics` ~0.2 с.

Метрики в `scripts/metrics.py` считаются **формально корректно** (micro-агрегация по совпавшим `itemId`, сравнение с `targetDetectedMcIds`). «Идеальность» **числовых** показателей на всём датасете без обучения на разметке недостижима; мы оптимизируем близость к эталону за счёт правил и словаря.

## 7. Ограничения текущей версии

- Остаются пограничные кейсы для `shouldSplit` и редкие пропуски фраз (синонимы, очень нестандартные формулировки).
- LLM улучшает **формулировки** черновиков, но не гарантирует совпадение эталона по категориям — за это отвечает эвристическое ядро.

## 9. Веб-интерфейс

После установки зависимостей:

```bash

```

Откройте в браузере `http://127.0.0.1:8000` — форма отправляет текст объявления на `POST /api/analyze` и показывает `detectedMcIds`, `shouldSplit`, `drafts` и полный JSON.

Тот же контракт доступен для интеграций без UI: тело запроса JSON с полями `description`, `mcId`, `mcTitle`, опционально `use_llm_drafts`.
