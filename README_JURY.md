# Avito Services Splitter — версия для жюри

## 1) Что решаем

В категории "Ремонт и отделка" одно объявление часто содержит несколько услуг, но публикуется в одной микрокатегории.  
Из-за этого исполнитель не попадает в точечные выдачи по другим релевантным услугам.

Наша система автоматически определяет:
- какие дополнительные микрокатегории реально присутствуют в тексте;
- нужно ли делать split (`shouldSplit`);
- какие черновики объявлений создать (`drafts`).

Ключевой принцип: **отделяем "услуга как часть комплекса" от "услуга как самостоятельное предложение"**.

---

## 2) Почему выбрали именно такое решение

Мы используем **гибрид**:

1. **Ядро (всегда):** детерминированный pipeline — словарь `keyPhrases` + эвристики контекста для `detectedMcIds` и `shouldSplit`.  
   Это даёт **контроль метрик**, скорость и объяснимость без GPU и без локального LLM.

2. **Опционально:** **удалённый LLM API** только для улучшения текстов в `drafts` (один батч-запрос на объявление).  
   Классификация не отдаётся в LLM — так мы сохраняем **воспроизводимость** и не ломаем автоматическую оценку.

### Причины выбора
- **Контроль качества:** микрокатегории и split трассируются до словаря и правил.
- **Скорость:** полный прогон по выданному JSON укладывается в **~2–3 с** на CPU без LLM.
- **Черновики:** при необходимости — человеко-подобный текст через API, без локального развёртывания моделей.
- **Масштабирование:** новая категория = в основном новый словарь + точечные правила.

Для целевого масштаба (до ~10 категорий, 5–15 микрокатегорий в каждой) это даёт лучший баланс:
**качество / скорость / управляемость**.

## 3) Архитектура решения

### Вход
- объявление: `itemId`, `sourceMcId` (или `mcId`), `description`;
- словарь микрокатегорий: `mcId`, `mcTitle`, `keyPhrases`.

### Шаг 1. Детекция `detectedMcIds`
- Нормализация текста.
- Поиск характерных `keyPhrases`.
- Фиксация всех найденных микрокатегорий.

### Шаг 2. Решение `shouldSplit`
- Для каждого совпадения анализируем окно контекста.
- Маркеры "самостоятельной услуги": `отдельно`, `самостоятельно`, и т.д.
- Маркеры "комплексной услуги": `под ключ`, `включая`, `в составе`, и т.д.
- В split попадают только самостоятельные услуги.
- Исходная микрокатегория объявления из результата исключается.

### Шаг 3. Черновики `drafts`
- Для каждой split-микрокатегории формируем объект:
  - `mcId`
  - `mcTitle`
  - `text` — шаблон или (при включённом API) улучшенная формулировка; поле может быть пустым.

---

## 4) Веб-интерфейс

- Запуск: `uvicorn web.app:app --host 127.0.0.1 --port 8000`
- Страница `GET /` — форма ввода объявления; `POST /api/analyze` — тот же JSON-ответ (`detectedMcIds`, `shouldSplit`, `drafts`), что и в задании заказчика.

---

## 5) Production-готовность

### Что уже сделано
- Четкие схемы входа/выхода (`pydantic`) и стабильный JSON-контракт.
- Веб-интерфейс (`web/app.py`) и REST `POST /api/analyze`.
- Отдельные CLI-скрипты:
  - `scripts/evaluate.py` — массовый прогон;
  - `scripts/run_local.py` — локальная проверка;
  - `scripts/metrics.py` — воспроизводимая оценка.
- Валидации данных и устойчивость к некорректным строкам датасета.
- Unit-тесты на ключевую логику.

### Соответствие ограничениям
- Локальные LLM не используются.
- Технические поля `caseType` и `split` в расчете не используются.
- Метрики категорий считаются по `targetDetectedMcIds`.
- Контролируется инвариант:
  - при `shouldSplit=true` должно выполняться  
    `targetDetectedMcIds == targetSplitMcIds`.

## 6) Результаты на rnc_dataset_markup

Метрики (эвристическое ядро, без LLM в прогоне):
- `precision_micro`: **0.966**
- `recall_micro`: **0.705**
- `f1_micro`: **0.815**
- `shouldSplit_accuracy`: **0.675**

Скорость:
- полный прогон `evaluate` (~2.5k объявлений): **~2–3 с** на CPU;
- расчет метрик: ~0.2 с.

Вывод:
- детекция и split близки к эталону на массиве; узкое место — редкие формулировки и пограничные split-кейсы;
- для продакшена качество черновиков повышается опциональным LLM-слоем без замедления классификации.

## 7) Масштабирование на новые категории

### Как тиражируется решение
1. Добавляем словарь новой категории (`mcId`, `keyPhrases`).
2. При необходимости уточняем 1–2 контекстных правила.
3. Запускаем тот же pipeline и те же метрики.

### Почему масштабирование недорогое
- Нет переобучения сложной модели при каждом изменении.
- Основная работа — редактура словаря, доступная и аналитикам, и разработчикам.
- Единый кодовый контур для всех категорий.

---

## 8) План улучшения (коротко)

1. Усилить правила для кейсов "перечисление услуг подряд" без явного "под ключ".
2. Расширить словарь с учетом словоформ/синонимов.
3. Ввести confidence-score и пороги принятия split-решения.
4. Добавить "golden set" спорных кейсов для регрессии качества.

---

## 9) Итог для бизнеса

Предложенное решение:
- быстрое и дешевое в эксплуатации;
- полностью контролируемое и объяснимое;
- легко раскатывается на смежные категории;
- уже дает высокий уровень точности детекции;
- готово к итеративному повышению качества split-решений без смены архитектуры.
